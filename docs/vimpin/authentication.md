# Authentication and rate limits

vimpin shells out to `git ls-remote` to resolve tags and branches — it
does **not** call the GitHub REST API. This has two practical
consequences.

## Private repos use your local credentials

Authentication piggybacks on local git. Private repos resolve through
whichever credential helper your shell already has configured:

- `git credential` with a stored token
- `gh auth setup-git`
- SSH keys (`ssh://git@github.com/...` URLs)
- macOS Keychain / Linux libsecret integrations

vimpin sets `GIT_TERMINAL_PROMPT=0` so missing credentials fail fast
rather than hanging on a password prompt. If a `vimpin run` invocation
exits with a `terminal prompts disabled` style error, the underlying git
remote needs credentials that are not configured.

## The 60 req/h limit does not apply

GitHub's git protocol endpoints (`git://`, `https://github.com/.../*.git`)
are **not subject** to the 60 req/h unauthenticated REST quota. So even
configs with several hundred plugins resolve cleanly from an
unauthenticated environment — useful for CI ephemeral runners, where a
PAT is not always available or desirable.

If you do want a token in CI (private repos, or simply to avoid
rate-limit noise on a shared egress IP), inject one through git's
credential mechanism rather than as an environment variable:

```bash
git config --global url."https://${GITHUB_TOKEN}:x-oauth-basic@github.com/".insteadOf "https://github.com/"
vimpin run --verify --check
```

## Hosts other than github.com

Hosts other than `github.com` are **not yet supported** in the CLI:
vimpin constructs `https://github.com/<owner>/<repo>` from each spec's
positional string.

Two workarounds while non-GitHub support is in progress:

1. **Local git aliases.** If your dotfiles include plugins from a
   self-hosted Gitea, mirror them on GitHub or use a `url.insteadOf`
   config to redirect:

   ```bash
   git config --global url."https://gitea.example.com/".insteadOf "https://github.com/mirrors/"
   ```

   Then specify the plugin as `mirrors/<repo>` in the lua spec.

2. **`-- vimpin:ignore`.** Skip the offending plugin entirely:

   ```lua
   { "owner/repo", url = "https://gitea.example.com/owner/repo" }, -- vimpin:ignore
   ```

Self-hosted forge support is an active area; track progress on the
[vimpin issue tracker](https://github.com/gr1m0h/vimpin/issues).

## What about `lazy-lock.json`?

vimpin does not read or write `lazy-lock.json`. The canonical Lua spec
is the source of truth. `lazy-lock.json` continues to be regenerated by
`:Lazy sync` from your spec, but plays no role in vimpin's pinning loop.

If you want Renovate to also manage `lazy-lock.json`, see the
[lazy-nvim-lock preset](../vimpin-renovate-config/lazy-lock.md) — it has
its own caveats (the lockfile entry must include a `url` field).
