# CI setup

The end-to-end recipe for gating a dotfiles repository on vimpin in GitHub
Actions. Suitable as a drop-in for an existing repo or as a reference for
a fresh one.

## What you'll wire up

1. A **required PR check** that fails on unpinned, malformed, or drifted
   specs.
2. A **scheduled update workflow** that opens a single weekly PR with
   plugin bumps.
3. (Optional) **Branch protection** so the PR check is enforced on
   `main`.

## 1. PR check workflow

`.github/workflows/vimpin.yml`:

```yaml
name: vimpin
on:
  pull_request:
    paths:
      - 'lua/**/*.lua'
      - 'init.lua'
      - 'plugin/**/*.lua'
  push:
    branches: [main]
    paths:
      - 'lua/**/*.lua'
      - 'init.lua'
      - 'plugin/**/*.lua'
permissions:
  contents: read
jobs:
  vimpin:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Cheapest gate first — offline structural check.
      - name: Structural check (no network)
        uses: gr1m0h/vimpin-action@v0.1.0
        with:
          mode: no-api

      # Anything that still needs canonicalisation? Fail closed.
      - name: Canonical-form check
        uses: gr1m0h/vimpin-action@v0.1.0
        with:
          mode: check

      # Annotations still aligned with their SHAs upstream?
      - name: Annotation-drift check
        uses: gr1m0h/vimpin-action@v0.1.0
        with:
          mode: verify-check
```

The three modes are deliberately layered cheapest-first: the
network-free check runs first; the network-bound drift check runs last.
On a 50-plugin dotfiles repo the entire workflow finishes in under a
minute.

## 2. Scheduled update workflow

`.github/workflows/vimpin-update.yml`:

```yaml
name: vimpin update
on:
  schedule:
    - cron: '0 9 * * 1'  # Mondays, 9am UTC
  workflow_dispatch:
permissions:
  contents: write
  pull-requests: write
jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Bump pinned specs
        id: vimpin
        uses: gr1m0h/vimpin-action@v0.1.0
        with:
          mode: update

      - name: Open PR
        if: steps.vimpin.outputs.changed == 'true'
        uses: peter-evans/create-pull-request@v6
        with:
          title: "chore(deps): vimpin --update"
          branch: vimpin-update
          commit-message: "chore(deps): bump pinned commits to latest tags"
          body: |
            Bumped pinned Lua specs to their latest semver tags
            (or branch HEADs for `-- branch:` annotated specs).
            Generated by `gr1m0h/vimpin-action` on a scheduled run.
          labels: |
            vimpin
            dependencies
```

This is the **batch** approach: one PR per week, all plugins. If you
prefer per-plugin PRs, see
[Renovate route](#alternative-renovate-instead-of-the-scheduled-workflow)
below — pick one, not both, or they'll race.

## 3. Branch protection

In **Settings → Branches → main → Branch protection rules**:

- ☑ Require status checks to pass before merging
- ☑ Require branches to be up to date before merging
- Add `vimpin / vimpin` as a required status check

That makes the PR check workflow a hard gate. A PR adding a new plugin
without running `vimpin run` first will be unmergeable until pinned.

## Alternative: Renovate instead of the scheduled workflow

If you'd rather have **one PR per plugin** (better hygiene, easier to
revert a single noisy update), use the
[`vimpin-renovate-config`](../vimpin-renovate-config/overview.md)
preset instead of the scheduled update workflow:

```json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": [
    "config:recommended",
    "github>gr1m0h/vimpin-renovate-config"
  ]
}
```

Disable the `vimpin-update.yml` workflow when going this route — both
mechanisms will otherwise try to open updates against the same plugins.

## What you don't need to wire

- **A nightly `verify` job.** `verify-check` runs on every PR; if no
  PRs are open, no drift can have landed.
- **`vimpin run` on `push: main`.** The PR check ensures `main` is
  always canonical; running `vimpin run` directly on main would be a
  no-op.
- **Caching of `actions/setup-go`.** The action installs a small Go
  toolchain; on `ubuntu-latest` the install is fast enough that caching
  rarely pays for itself.

## Verifying the setup

Open a PR that adds a plugin without a pin:

```lua
return {
  { "ggandor/leap.nvim", tag = "v0.1.5" },
  { "folke/snacks.nvim" },  -- intentionally unpinned
}
```

You should see:

- `Structural check (no network)` ✗ — fails fast on the missing
  `commit` field.
- The PR is blocked from merging by the required status check.

Fix it with `vimpin run` locally, push, and the same workflow goes
green.
