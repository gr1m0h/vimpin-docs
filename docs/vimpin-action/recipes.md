# Recipes

Copy-paste-ready workflows for the most common scenarios.

## Required check on every PR

The standard PR-gate. Three orthogonal modes in one workflow, each
catching a different failure mode.

```yaml
# .github/workflows/vimpin.yml
name: vimpin
on:
  pull_request:
    paths:
      - 'lua/**/*.lua'
      - 'init.lua'
permissions:
  contents: read
jobs:
  vimpin:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: gr1m0h/vimpin-action@v0.1.0
        with:
          mode: check
      - uses: gr1m0h/vimpin-action@v0.1.0
        with:
          mode: no-api
      - uses: gr1m0h/vimpin-action@v0.1.0
        with:
          mode: verify-check
```

Make `vimpin / vimpin` a required status check on `main` in your
repository settings.

## Scheduled update PR

A `workflow_dispatch` / cron workflow that bumps every pinned spec to
the latest semver tag (or current branch HEAD) and opens a single PR
with the result.

```yaml
# .github/workflows/vimpin-update.yml
name: vimpin update
on:
  schedule:
    - cron: '0 9 * * 1'   # Mondays, 9am UTC
  workflow_dispatch:
permissions:
  contents: write
  pull-requests: write
jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: gr1m0h/vimpin-action@v0.1.0
        id: vimpin
        with:
          mode: update
      - if: steps.vimpin.outputs.changed == 'true'
        uses: peter-evans/create-pull-request@v6
        with:
          title: "chore(deps): vimpin --update"
          branch: vimpin-update
          commit-message: "chore(deps): bump pinned commits to latest tags"
          body: |
            Bumped pinned Lua specs to their latest semver tags
            (or branch HEADs for `-- branch:` annotated specs).
            Generated by `gr1m0h/vimpin-action` on a scheduled run.
```

The PR will land for review with both the commit hash and the
annotation rewritten atomically, in canonical form. After merge, the
required-check workflow above continues to gate future PRs.

## Cron without a PR (auto-merge)

If you trust vimpin's `--update` enough to auto-merge in a
non-production context (e.g., a personal dotfiles repo), skip the
`create-pull-request` step and push directly:

```yaml
jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: gr1m0h/vimpin-action@v0.1.0
        id: vimpin
        with:
          mode: update
      - if: steps.vimpin.outputs.changed == 'true'
        run: |
          git config user.name "vimpin-bot"
          git config user.email "vimpin-bot@users.noreply.github.com"
          git add -A
          git commit -m "chore(deps): vimpin --update"
          git push
```

Skip this pattern if `main` is protected — it will fail at the push.
Use the PR-creating recipe instead.

## Combined with Renovate

You usually want **either** the scheduled update workflow **or**
Renovate's [vimpin preset](../vimpin-renovate-config/overview.md), not
both — they'd race to open the same PRs.

The split that works in practice:

- **Renovate** for per-plugin granular PRs (one PR per plugin update).
  Better PR hygiene; easier to revert a single noisy plugin.
- **`mode: update`** for batched weekly updates of every plugin in one
  PR. Less PR noise; faster to review when you trust everything in
  scope.

Pick whichever matches your update tolerance, and disable the other.

## Self-hosted runners

The action's defaults assume `ubuntu-latest`. On a self-hosted runner,
verify that:

- The `actions/setup-go` step can install Go 1.24 (or override
  `go-version` to a version already present on the runner).
- `git` is on `$PATH`.
- The runner can reach `github.com` over HTTPS — vimpin shells out to
  `git ls-remote https://github.com/...`.

No additional configuration is required on the action side.
